\chapter{Testfälle und Testszenarien}
\section{Tests}
\begin{description}
	\tf Die GUI reagiert darauf, wenn ein Knoten maßgeblich von seinen Sollwerten abweicht z.B. zur eine Einfärbung des problematischen Knotens.
	\tf Eingelesene Konfigurationsdaten lassen sich parsen und auslesen
	\tf Hostdaten werden korrekt übermittelt
	\tf rqt Plugins reagieren korrekt auf Großenänderung
	\tf Optional: Funktionsweise des Starten und Stoppens von Knoten unter Netzwerklast prüfen
	\tf Menge der Metadaten überschreitet nicht die Netzwerkkapazität
\end{description}

\section{Beispielszenarien}

\subsection*{Typische Vorgehensweise}
\begin{itemize}
	\item Der Benutzer startet zunächst roscore, dann neben den von ihm benötigten Knoten den Monitoring Knoten
	\item Um den aktuellen Status seiner Knoten und des Netzwerks zu überwachen, startet er nun rqt und öffnet das Listen Widget. Falls nun Fehler auftreten wird er durch eine farbliche Veränderung auf mögliche Probleme hingewiesen.
\end{itemize}

\subsection*{Knoten exisitert nicht mehr}
\begin{itemize}
  \item Ein ROS-Netzwerk mit beliebig vielen Knoten wird gestartet
  \item Der Benutzer öffnet das Plugin in der rqt GUI
  \item Nun wird ein Knoten vom Netzwerk getrennt (z.B. durch Programmabsturz)
  \item	Falls dieser Knoten in den Soll-Spezifkationen als wichtig definiert wurde, reagiert die GUI nun darauf reagieren und zeigt an, dass der Knoten nicht mehr
  verfügbar ist und dass ein Fehler aufgetreten ist
  \item Optional bietet die GUI nun die Möglichkeit den betroffenen Knoten neuzustarten
\end{itemize}
